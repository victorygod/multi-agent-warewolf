## 线下 vs 在线 核心差异

| 维度 | 线下法官 | 在线服务器 |
|-----|---------|-----------|
| **身份确认** | 法官让玩家睁眼确认 | 服务器推送私有状态到客户端，玩家在自己的设备上查看 |
| **夜晚行动** | 法官逐个唤醒，玩家手势比划 | 客户端UI操作，狼人群聊打字 |
| **情侣确认** | 法官让情侣睁眼互相看 | 服务器只推送"你们是情侣"，不告知身份/阵营，丘比特也不知道情侣身份 |
| **丘比特** | 丘比特知道情侣身份 | 丘比特不知道情侣身份，只知道座位号 |
| **发言** | 法官指定顺序，轮流发言 | 系统广播发言，实时推送 |
| **投票** | 玩家举手/指人 | 前端按钮点击 |
| **闭眼/睁眼** | 法官喊话 | 客户端UI状态切换 |
| **状态记录** | 纸笔记录 | 服务器内存 |
| **决策辅助** | 查阅文档 | 代码实现决策树 |

## 在线系统设计原则

### 1. 纯自动推进
- 服务器按照状态机自动推进游戏阶段
- 管理员仅在极端情况介入（踢人、重开）
- 所有规则判定由代码自动执行

### 2. 状态驱动
- 服务器维护完整游戏状态（玩家、阵营、存活、技能状态）
- 每个阶段有明确的入口条件和出口条件
- 阶段转换自动触发，无需人工干预

### 3. 实时通信
- WebSocket推送，实时同步状态
- 广播机制：发言、投票、死亡结果全员可见
- 私有消息：夜间狼人聊天仅狼人可见

### 4. 客户端交互
- 玩家通过客户端UI进行操作（选择目标、投票、发言）
- 服务器接收操作请求，验证合法性，执行操作

| 功能 | 线下做法 | 在线实现 |
|-----|---------|---------|

| **女巫自救限制** | 法官口头阻止 | 前端直接禁用自救选项 |

| **胜负判定** | 法官宣布 | 服务器自动检测并推送结果 |
| **票数计算** | 法官手动计算 | 服务器自动计算（警长1.5票） |

## 游戏阶段总览

```
游戏开始
    │
    ▼
发牌阶段 → 服务器分配身份，推送私有状态
    │
    ▼
第一夜特殊阶段 → 丘比特连接情侣
    │
    ▼
夜晚阶段（循环）
    │
    ├── 守卫守护
    ├── 狼人（群聊+刀杀）
    ├── 女巫用药
    ├── 预言家查验
    └── 死亡结算（自动）
    │
    ▼
白天阶段（循环）
    │
    ├── 警长竞选（仅第一天，最先进行）
    ├── 天亮公布死亡
    ├── 遗言处理（首夜死亡玩家）
    ├── 白天讨论（顺序发言）
    ├── 放逐投票（按钮）
    └── 放逐后处理（猎人/白痴翻牌）
    │
    ▼
胜负判定 → 推送结果
```

## 核心自动化逻辑

### 自动结算（对照线下文档的关键规则）

| 规则 | 线下法官操作 | 在线服务器自动逻辑 |
|-----|------------|------------------|
| **同守同救** | 法官口播"死亡" | 服务器检测到守卫守护目标=女巫解药目标 → 该玩家死亡 |
| **毒杀** | 法官记录毒杀 | 服务器检测到毒药命中 → 死亡（不受守护影响） |
| **殉情** | 法官判定情侣 | 服务器检测到情侣一方死亡 → 另一方死亡 |
| **遗言权限** | 法官判断给不给 | 服务器检测：首夜死亡或白天放逐 → 有遗言；否则无 |
| **猎人翻牌** | 法官宣布翻牌 | 服务器检测：猎人被刀/被放逐 → 翻牌，可以开枪 |
| **白痴翻牌** | 法官宣布翻牌 | 服务器检测：白痴被放逐 → 翻牌免疫；被刀/被毒 → 直接死亡 |
| **胜负判定** | 法官宣布 | 服务器自动检测胜利条件并推送结果 |

## 丘比特情侣信息推送规则

**线下做法**：法官让情侣睁眼互相看，双方知道对方身份

**在线做法**：
- 丘比特不知道情侣的具体身份，只知道座位号
- 情侣只知道对方是情侣，**不知道**对方的身份和阵营
- 服务器只推送："你的情侣是X号玩家"

---

## 翻牌规则

**线下做法**：猎人/白痴死亡时翻牌

**在线做法**：
- 猎人：被刀杀或被放逐时翻牌，可以开枪
- 白痴：被放逐时翻牌，免疫放逐
- 其他所有角色：死亡时不翻牌（只在遗言时说身份）
